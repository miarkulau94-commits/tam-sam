# Логика исполнения ордеров в AutoScaleX Pro 2.2

**Шаг сетки** — мультипликативный: между соседними уровнями примерно 1.5% (или 0.75%). Каждый следующий уровень = предыдущий × (1 ± grid_step_pct). После исполнения BUY создаётся **только** SELL; после исполнения SELL создаётся BUY снова на том же уровне (ниже цены продажи на шаг).

---

## Общий цикл проверки

**Главный цикл (`main_loop`)**:
1. Каждые 15 секунд (в состоянии `TRADING`)
2. **ВАЖНО: порядок вызовов**:
   - Сначала: `check_orders()` - проверка исполнения
   - Потом: `sync_orders_from_exchange()` - синхронизация
   - Затем: `check_critical_situation()` - проверка критических ситуаций

## Обнаружение исполненных ордеров

**Функция `check_orders()`**:
1. Получает список открытых ордеров на бирже через API
2. Получает список ордеров в памяти со статусом `'open'`
3. **Сравнивает**: если ордер есть в памяти как `'open'`, но его нет в списке открытых на бирже → ордер исполнился
4. Для каждого исполненного ордера вызывает соответствующую функцию обработки

## Обработка исполнения BUY ордера

**Функция `handle_buy_filled(order, price)`**:

### Шаг 1: Обновление данных
- Рассчитывает реальное количество после комиссии: `btc_received = order.qty * (1 - FEE_RATE)`
- Увеличивает счетчик: `total_executed_buys += 1`
- Добавляет позицию в `position_manager`
- Сохраняет сделку в статистику
- Обновляет статус ордера: `status = 'filled'`

### Шаг 2: Создание SELL ордера (после BUY создаётся только SELL)
- **Цена**: `sell_price = buy_price * (1 + grid_step_pct)` — выше на шаг сетки (мультипликативно)
- **Количество**: использует `btc_received` (после комиссии)
- Проверки:
  - Достаточно базового актива
  - Нет ордера на этой цене
  - Количество >= min_qty
  - Номинал >= min_notional
- Новый BUY на шаг ниже при исполнении BUY в основном потоке **не создаётся**. Создаётся только SELL.

### Шаг 3: Синхронизация и дополнительные проверки
- Синхронизирует ордера с биржей
- Проверяет критический уровень
- Проверяет пирамидирование (если profit_bank > 0)
- Сохраняет состояние

## Обработка исполнения SELL ордера

**Функция `handle_sell_filled(order, price)`**:

### Шаг 1: Обновление данных
- Рассчитывает прибыль через `position_manager.calculate_profit_for_sell()`
- Увеличивает `profit_bank` на прибыль
- Увеличивает счетчик: `total_executed_sells += 1`
- Сохраняет сделку в статистику
- Обновляет статус ордера: `status = 'filled'`

### Шаг 2: Создание BUY ордера (СТРОГО ТОЛЬКО ЭТО!)
- Вызывает `create_buy_after_sell(sell_price)`
- **Цена**: `new_buy_price = sell_price * (1 - grid_step_pct)` - строго ниже на шаг сетки
- **Количество**: `qty = buy_order_value / new_buy_price`
- Проверки:
  - Достаточно USDT (quote_balance >= buy_order_value)
  - Не превышен лимит BUY ордеров (MAX_BUY_ORDERS)
  - Нет ордера на этой цене
  - Количество >= min_qty
  - Номинал >= min_notional
- **Создает ТОЛЬКО BUY ордер ниже исполненного SELL на шаг сетки**

### Шаг 3: Синхронизация и дополнительные проверки
- Синхронизирует ордера с биржей
- Проверяет пирамидирование (если profit_bank > 0)
- Проверяет ребалансировку (`check_rebalancing`)
- Сохраняет состояние

## Синхронизация ордеров

**Функция `sync_orders_from_exchange()`**:
1. Получает список открытых ордеров на бирже
2. **Удаляет из памяти** ордера, которых нет на бирже (статус != 'open' остается)
3. **Добавляет в память** ордера, которые есть на бирже, но нет в памяти

**Важно**: Эта функция удаляет исполненные ордера из памяти, поэтому она должна вызываться ПОСЛЕ `check_orders()`!

## Ключевые моменты

1. **Порядок проверки критичен**: `check_orders()` → `sync_orders_from_exchange()`
2. **Обнаружение исполнения**: Сравнение списка ордеров в памяти (статус 'open') со списком открытых на бирже
3. **СТРОГО: После BUY**: Создается ТОЛЬКО SELL выше на шаг сетки (`price * (1 + grid_step_pct)`)
4. **СТРОГО: После SELL**: Создается ТОЛЬКО BUY ниже на шаг сетки (`price * (1 - grid_step_pct)`)
5. **Сетка поддерживается**: Ордера всегда создаются на расстоянии `grid_step_pct` друг от друга
6. **Проверки при создании**: Баланс, лимиты, валидация (min_qty, min_notional), существующие ордера

## Возможные проблемы

1. **Ордера не обнаруживаются**: Если `sync_orders_from_exchange()` вызывается до `check_orders()`, исполненные ордера удаляются из памяти до обработки
2. **Новые ордера не создаются**: Может быть недостаточно баланса, превышен лимит ордеров, или не пройдена валидация
3. **Дублирование ордеров**: Проверяется существование ордера на той же цене (с точностью до tick)
