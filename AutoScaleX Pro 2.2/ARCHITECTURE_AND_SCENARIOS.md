# Архитектура и сценарии (AutoScaleX Pro 2.2)

Документ описывает высокоуровневую архитектуру бота и ключевые сценарии: ребаланс, подготовка к ребалансу (1 SELL → отмена 5 BUY), восстановление 5 BUY внизу после рестарта, защита сетки.

Подробности по шагам и формулам — в `BOT_LOGIC.md`, `SELL_REBALANCING.md`, `GRID_PROTECTION.md`, `ORDER_EXECUTION_LOGIC.md`, `GRID_FREE_LEVELS.md` (свободный уровень при занятой цене).

**Шаг сетки:** везде мультипликативный (~1.5% или 0.75% между уровнями). После исполнения BUY создаётся только SELL; после исполнения SELL — BUY снова на том же уровне (см. `ORDER_EXECUTION_LOGIC.md`, `BOT_LOGIC.md` §2.5).

---

## 1. Архитектура

### 1.1 Основные компоненты

| Компонент | Назначение |
|-----------|------------|
| **TradingBot** | Ядро: состояние (TRADING/PAUSED/STOPPED), ордера в памяти, позиции FIFO, логика обработки исполнений и ребаланса |
| **BingXSpot / BingXSpotAsync** | Работа с API BingX: ордера, балансы, тикер. Глобальный лимитер 40 RPS, ретраи при 429/5xx. **Circuit breaker** не открывается из‑за одних только ошибок rate limit (429 / превышение лимита) |
| **Persistence** | Сохранение/загрузка state (JSON по user_id), ордера, настройки пользователя, сделки |
| **PositionManager** | FIFO-учёт позиций для расчёта прибыли при SELL |
| **Telegram** | Интерфейс пользователя: старт/стоп, настройки, уведомления |

### 1.2 Главный цикл (main_loop)

В состоянии **TRADING** каждые ~15 секунд:

1. **check_orders()** — сравнение ордеров в памяти (status=open) с биржей; если ордера нет на бирже — считаем исполненным → **handle_buy_filled** или **handle_sell_filled**.
2. **sync_orders_from_exchange()** — приведение списка ордеров в памяти в соответствие с биржей. Источник истины по «что открыто» — ответ **`open_orders`**. Для id, которые в памяти open, но отсутствуют в этом списке: не более **`SYNC_GET_ORDER_MAX_PER_CALL`** уточнений через **`get_order`** за один проход (сначала SELL, затем BUY); остальные обрабатываются как исполнение по цене из памяти (как в п.1). При открытии **«Баланс»** в Telegram используется **`SYNC_BALANCE_MAX_GET_ORDER`** вместо полного лимита.
3. **check_critical_situation()** — все BUY исполнены, нет SELL, не хватает USDT на рыночную покупку.
4. **check_protection_add_five_buy_when_three_left()** — защита «3 BUY → до 5 BUY внизу» при большой сетке.

Исполнение ордера обрабатывается в момент обнаружения в **check_orders**; отмена 5 BUY при 1 SELL выполняется внутри **handle_sell_filled**, а не в цикле.

### 1.3 Сохраняемое состояние (state)

- `uid`, `symbol`, `grid_step_pct`, `buy_order_value`, `profit_bank`, `initial_equity`, счётчики сделок.
- `orders` — только открытые ордера (open).
- `bot_state` — TRADING / PAUSED / STOPPED для авто-восстановления после рестарта.
- **cancelled_buy_for_rebalance_prep** — флаг «отменили 5 BUY при 1 SELL»; после рестарта позволяет восстановить 5 BUY внизу, когда снова станет ≥3 SELL.

## 2. Сценарии

### 2.1 Ребалансировка (все SELL исполнены)

**Триггер:** В памяти и на бирже 0 открытых SELL (последний SELL только что исполнился).

**Где:** `check_rebalancing(current_price)` вызывается из **handle_sell_filled** после синхронизации ордеров.

**Последовательность:**

1. Проверка: `open_sell_orders == 0`, `state == TRADING`.
2. Проверка баланса: `quote_balance >= (5 × buy_order_value) + 2` USDT.
3. Рыночная покупка на сумму `(5 × buy_order_value) + 2` USDT.
4. Ожидание ~3 с, добавление позиции в FIFO.
5. **create_sell_grid_only()** — создание 5 SELL ордеров выше текущей цены с шагом сетки.
6. Если создано ≥3 SELL — **create_buy_orders_at_bottom()**: до 5 BUY внизу сетки (опционально).
7. Сброс флага **cancelled_buy_for_rebalance_prep** в `False`, сохранение state.

**Итог:** Появились 5 новых SELL и при необходимости BUY внизу; цикл продолжается.

Подробно: `SELL_REBALANCING.md`, `BOT_LOGIC.md` (§6.1).

---

### 2.2 Подготовка к ребалансу: 1 SELL → отмена 5 самых низких BUY

**Триггер:** После исполнения SELL ордера, после синхронизации с биржей открытых SELL осталось **ровно 1**.

**Где:** В **handle_sell_filled()** после `sync_orders_from_exchange()`.

**Последовательность:**

1. Условие: `len(open_sell_after) == 1`.
2. **cancel_last_n_buy_orders(5)** — отбор открытых BUY по возрастанию цены, отмена 5 с наименьшей ценой на бирже и удаление их из списка ордеров бота.
3. Установка флага **\_cancelled_buy_for_rebalance_prep = True**.
4. Инвалидация кеша баланса, при необходимости пауза ~1 с.
5. При следующем **save_state()** флаг сохраняется в state как **cancelled_buy_for_rebalance_prep: true**.

**Зачем:** Освободить USDT под будущую рыночную покупку при ребалансе, когда исполнится последний SELL.

**Итог:** USDT разблокирован; флаг записан в state и переживёт рестарт.

Подробно: `GRID_PROTECTION.md` (§2).

---

### 2.3 Восстановление 5 BUY внизу (после отмены 5 BUY при 1 SELL)

**Триггер:** Исполнился BUY ордер; после синхронизации открытых SELL **≥ 3** и флаг **\_cancelled_buy_for_rebalance_prep == True** (например, восстановлен из state после рестарта).

**Где:** В **handle_buy_filled()** после `check_rebalancing_after_all_buy_filled()`.

**Последовательность:**

1. Условие: `len(open_sell_after) >= 3` и `_cancelled_buy_for_rebalance_prep is True`.
2. **create_buy_orders_at_bottom(price)** — создание до 5 BUY ордеров ниже самой низкой текущей BUY-цены.
3. Сброс флага **\_cancelled_buy_for_rebalance_prep = False**.
4. При следующем **save_state()** флаг сохраняется как `false`.

**Смысл:** После того как при 1 SELL отменили 5 BUY, цена могла пойти вниз и не дойти до ребаланса. Когда снова накопилось ≥3 SELL, сетку внизу восстанавливаем, выставив до 5 BUY под текущей сеткой.

**После рестарта:** Флаг **cancelled_buy_for_rebalance_prep** загружается из state в **load_state()**. Пока не выполнится условие «BUY FILL + ≥3 SELL» или не пройдёт полный ребаланс, флаг остаётся; при срабатывании восстановления 5 BUY или по завершении ребаланса флаг сбрасывается и сохраняется.

Подробно: `GRID_PROTECTION.md` (§2), логика флага в state — см. сохранение/загрузку в `trading_bot.py`.

---

### 2.4 Защита «3 BUY → добавить до 5 BUY внизу»

**Триггер:** В основном цикле (TRADING): открытых BUY **≤ 3**, а общее число открытых ордеров (BUY + SELL) **больше порога**.

**Пороги:**

- Шаг 1.5%: открытых ордеров > **62**
- Шаг 0.75%: открытых ордеров > **127**

**Где:** **check_protection_add_five_buy_when_three_left()** вызывается из главного цикла после sync и check_critical_situation.

**Последовательность:**

1. Условия: `open_buy_count ≤ 3`, `open_buy_count + open_sell_count > threshold`.
2. **create_buy_orders_at_bottom(current_price)** — до 5 новых BUY ниже самой низкой текущей BUY-цены (с учётом лимита BUY и баланса USDT).

**Зачем:** При большой сетке и малом числе оставшихся BUY сетка «висит» высоко; защита заранее растягивает её вниз и снижает риск быстрого перехода в ребаланс при падении цены.

Подробно: `GRID_PROTECTION.md` (§1).

## 3. Сводка сценариев

| Сценарий | Триггер | Действие | Флаг / State |
|----------|---------|----------|--------------|
| Ребаланс | 0 SELL | Рыночная покупка + 5 SELL + опционально 5 BUY внизу | Сброс **cancelled_buy_for_rebalance_prep** |
| Подготовка к ребалансу | 1 SELL после SELL fill | Отмена 5 самых низких BUY | Установка **cancelled_buy_for_rebalance_prep = True**, сохранение в state |
| Восстановление 5 BUY внизу | BUY fill + ≥3 SELL + флаг True | create_buy_orders_at_bottom | Сброс флага, сохранение в state |
| Защита сетки | ≤3 BUY, большая сетка | create_buy_orders_at_bottom | Не использует флаг |
