# Логика работы бота AutoScaleX

## 📋 Общая архитектура

Бот работает в асинхронном режиме и использует три основные стратегии:
- **Grid Trading** - автоматическая сетка ордеров
- **Pyramiding** - расширение позиций через реинвестирование прибыли
- **DCA** - усреднение цены входа при падении

---

## 🚀 Этап 1: Инициализация бота

### 1.1 Запуск бота
1. Пользователь нажимает "▶️ Старт" в Telegram
2. Бот проверяет наличие API ключей
3. Если ключи не настроены → запрос на ввод через команду `/set_api_keys`
4. Если ключи есть → проверка их валидности

### 1.2 Проверка ключей и баланса
1. Создается временный экземпляр `BingXSpot` для проверки
2. Проверяется доступность торговой пары
3. Получаются балансы:
   - Quote asset (USDT) - для покупок
   - Base asset (BTC/ETH/SOL и т.д.) - для продаж
4. Рассчитывается общий equity (баланс в USDT)

### 1.3 Создание экземпляра TradingBot
1. Инициализируется `TradingBot` с API ключами
2. Загружаются сохраненные настройки:
   - Торговая пара (symbol)
   - Шаг сетки (0.75% или 1.5%)
   - Размер ордера (10/20/50/75/100/250 USDT)
3. Сбрасывается circuit breaker
4. Состояние бота: `INITIALIZING`

## 🎯 Этап 2: Создание сетки ордеров (create_grid)

### 2.1 Подготовка
1. Отменяются все существующие ордера на бирже
2. Очищается список ордеров в памяти
3. Получается информация о символе:
   - `stepSize` - минимальный шаг количества
   - `tickSize` - минимальный шаг цены
   - `minQty` - минимальное количество
   - `minNotional` - минимальный номинал ордера

### 2.2 ШАГ 1: Рыночная покупка для SELL ордеров
**Цель:** Приобрести базовый актив для создания SELL ордеров

1. Рассчитывается сумма покупки:
   ```
   market_buy_amount = (5 × buy_order_value) + 2 USDT
   Пример: (5 × 10) + 2 = 52 USDT
   ```
   - 5 ордеров × размер ордера = базовая сумма
   - +2 USDT = резерв для точного размещения ордеров

2. Проверяется баланс:
   - Если `quote_balance >= market_buy_amount` → выполняется рыночная покупка
   - Если баланса недостаточно → пропускается этот шаг

3. Рыночная покупка:
   - Используется `quoteOrderQty` (покупка на сумму, а не на количество)
   - После покупки ждем 3 секунды для исполнения
   - Получаем информацию об исполненном ордере
   - Добавляем позицию в `PositionManager` (FIFO учет)

4. Обновляется баланс базового актива

### 2.3 ШАГ 2: Создание SELL ордеров
**Цель:** Создать 5 SELL ордеров выше текущей цены

1. Проверяется доступный баланс базового актива:
   ```
   available_base = current_base_balance - locked_base_asset
   ```
   - `locked_base_asset` = сумма всех открытых SELL ордеров

2. Рассчитывается количество на ордер:
   ```
   sell_qty_per_order = available_base / 5
   ```

3. Создаются 5 SELL ордеров:
   - **Первый ордер:** `price × (1 + grid_step_pct)`
   - **Второй ордер:** `первый_ордер × (1 + grid_step_pct)`
   - И так далее...
   - Каждый ордер размещается на шаг сетки выше предыдущего

4. Проверки перед размещением:
   - `qty >= minQty`
   - `notional >= minNotional` (или 0, если биржа не требует)
   - Цена округляется до `tickSize`
   - Количество округляется до `stepSize`

### 2.4 ШАГ 3: Создание BUY ордеров
**Цель:** Создать сетку BUY ордеров ниже текущей цены

1. Рассчитывается количество BUY ордеров:
   ```
   buy_count = min(
       (quote_balance / buy_order_value) + (profit_bank / buy_order_value),
       MAX_BUY_ORDERS (130)
   )
   ```
   - Базовое количество = баланс / размер ордера
   - Дополнительные = прибыль / размер ордера
   - Максимум = 130 ордеров

2. Создаются BUY ордера:
   - **Первый ордер:** `price × (1 - grid_step_pct)`
   - **Второй ордер:** `первый_ордер × (1 - grid_step_pct)`
   - И так далее...
   - Каждый ордер размещается на шаг сетки ниже предыдущего

3. Для каждого ордера:
   - Рассчитывается количество: `qty = buy_order_value / price`
   - Проверяются ограничения (`minQty`, `minNotional`)
   - Размещается лимитный ордер на бирже

4. Результат:
   - Создано N BUY ордеров (от 1 до 130)
   - Создано 5 SELL ордеров
   - Состояние бота: `TRADING`

## 🔄 Этап 3: Основной цикл работы (main_loop)

### 3.1 Состояния бота

#### INITIALIZING
- Создается сетка ордеров
- После успешного создания → `TRADING`
- При ошибке → повтор через 60 секунд

#### TRADING (основное состояние)
1. **Синхронизация ордеров** (`sync_orders_from_exchange`):
   - Получаются все открытые ордера с биржи
   - Сравниваются с локальным списком
   - Удаляются ордера, которых нет на бирже
   - Добавляются новые ордера с биржи

2. **Проверка исполнения** (`check_orders`):
   - Для каждого открытого ордера проверяется статус на бирже
   - Если ордер исполнен → вызывается обработчик:
     - `handle_buy_filled()` для BUY
     - `handle_sell_filled()` для SELL

3. **Проверка критической ситуации** (`check_critical_situation`):
   - Все BUY исполнены
   - SELL ордера не созданы
   - Недостаточно баланса для рыночной покупки
   → Отправляется критическое уведомление

#### PAUSED
- Бот приостановлен
- Ордера остаются на бирже
- Можно возобновить через "▶️ Старт"

#### CRITICAL
- Все BUY ордера исполнены
- Создается критическая SELL сетка от VWAP
- Требуется пополнение баланса

### 3.2 Интервал проверки
- Каждые 15 секунд выполняется цикл
- Асинхронная обработка не блокирует другие операции

## 💰 Этап 4: Обработка исполненных ордеров

### 4.1 Исполнение BUY ордера (handle_buy_filled)

**Последовательность действий:**

1. **Обновление балансов:**
   ```
   base_asset = balance(base_asset_name)
   current_deposit = balance(quote_asset_name)
   ```

2. **Добавление позиции в FIFO:**
   ```
   btc_received = order.qty × (1 - fee_rate)
   position_manager.add_position(price, btc_received)
   ```
   - Позиция добавляется с временной меткой
   - Используется для расчета прибыли по методу FIFO

3. **Сохранение статистики:**
   - Тип: BUY
   - Цена, количество, сумма
   - Прибыль = 0 (покупка)

4. **Создание SELL ордера:**
   - Проверяется доступный баланс базового актива
   - Рассчитывается цена: `price × (1 + grid_step_pct)`
   - Рассчитывается количество из доступного баланса
   - Размещается лимитный SELL ордер
   - **Важно:** SELL создается сразу после BUY

5. **Проверка критического уровня:**
   - Если все BUY ордера исполнены → переход в `CRITICAL`
   - Создается критическая SELL сетка

6. **Проверка Pyramiding:**
   - Если `profit_bank >= buy_order_value` → создается новый BUY ордер

### 4.2 Исполнение SELL ордера (handle_sell_filled)

**Последовательность действий:**

1. **Расчет прибыли по FIFO:**
   ```
   profit = position_manager.calculate_profit_for_sell(
       sell_qty, 
       sell_price, 
       fee_rate
   )
   ```
   - Используется метод FIFO (First In, First Out)
   - Берется самая старая позиция из очереди
   - Рассчитывается: `(revenue - cost) - fees`

2. **Обновление profit_bank:**
   ```
   profit_bank += profit
   ```
   - Прибыль накапливается в "банке прибыли"
   - Используется для Pyramiding

3. **Сохранение статистики:**
   - Тип: SELL
   - Цена, количество, сумма
   - Рассчитанная прибыль
   - Текущий profit_bank

4. **Проверка Pyramiding:**
   - Если `profit_bank >= buy_order_value` → создается новый BUY

5. **Проверка ребалансировки:**
   - Если все SELL ордера исполнены → выполняется ребалансировка

## 📈 Этап 5: Стратегии

### 5.1 Grid Trading (Сеточная торговля)

**Принцип:**
- Создается сетка ордеров выше и ниже текущей цены
- При движении цены ордера автоматически исполняются
- Прибыль фиксируется на каждом шаге сетки

**Параметры:**
- Шаг сетки: 0.75% или 1.5%
- Количество BUY: от 1 до 130 (зависит от баланса)
- Количество SELL: 5 ордеров

**Преимущества:**
- Работает в боковом тренде
- Автоматическое усреднение при падении
- Фиксация прибыли при росте

### 5.2 Pyramiding (Пирамида)

**Принцип:**
- При накоплении прибыли создаются новые BUY ордера
- Новые ордера размещаются ниже самой низкой цены в сетке
- Расширяет сетку вниз для большего усреднения

**Алгоритм:**
```
while profit_bank >= buy_order_value:
    1. Находим самый низкий открытый BUY ордер
    2. Рассчитываем новую цену: lowest_price × (1 - grid_step_pct)
    3. Создаем новый BUY ордер на эту цену
    4. Вычитаем из profit_bank: buy_order_value
    5. Повторяем, пока есть прибыль и место (макс. 125 BUY при 0.75%, 60 BUY при 1.5%)
```

**Условия:**
- `profit_bank >= buy_order_value`
- Количество открытых BUY < максимума (125 при шаге 0.75%, 60 при шаге 1.5%)
- Есть доступный баланс USDT

**Результат:**
- Сетка расширяется вниз
- Больше возможностей для усреднения
- Увеличение потенциальной прибыли

### 5.3 DCA (Dollar Cost Averaging)

**Принцип:**
- При падении цены исполняются BUY ордера
- Каждый новый BUY исполняется по более низкой цене
- Средняя цена покупки снижается

**Реализация:**
- Автоматически работает через Grid Trading
- При падении цены исполняются BUY ордера по порядку
- Каждый BUY добавляется в FIFO очередь
- При продаже используется средняя цена из FIFO

**Преимущества:**
- Снижение средней цены входа
- Уменьшение рисков при волатильности
- Автоматическое усреднение без ручного вмешательства

## 🔄 Этап 6: Ребалансировка

### 6.1 Автоматическая ребалансировка (check_rebalancing)

**Триггер:** Все SELL ордера исполнены

**Алгоритм:**

1. **Проверка условий:**
   ```
   if len(open_sell_orders) == 0 and state == TRADING:
   ```

2. **Рыночная покупка:**
   ```
   market_buy_amount = (5 × buy_order_value) + 2 USDT
   ```
   - Покупка на фиксированную сумму
   - Используется `quoteOrderQty` (покупка на сумму)
   - +2 USDT резерв для точного размещения

3. **Ожидание исполнения:**
   - Ждем 3 секунды
   - Получаем информацию об ордере
   - Добавляем позицию в FIFO

4. **Создание новых SELL ордеров:**
   - Вызывается `create_sell_grid_only()`
   - Создается 5 SELL ордеров
   - Первый ордер: `current_price × (1 + grid_step_pct)`
   - Последующие: каждый на шаг сетки выше

5. **Результат:**
   - Приобретен базовый актив
   - Создана новая SELL сетка
   - Цикл продолжается

### 6.2 Ручная ребалансировка SELL

**Команда:** "🔄 Ребалансировать SELL"

**Алгоритм:**
1. Отменяются все существующие SELL ордера
2. Рассчитывается VWAP (средняя цена покупки)
3. Создается 3 SELL ордера от VWAP:
   - `VWAP × 1.02` (2% прибыль)
   - `VWAP × 1.04` (4% прибыль)
   - `VWAP × 1.06` (6% прибыль)
4. Весь доступный баланс распределяется равномерно

## ⚠️ Этап 7: Критические ситуации

### 7.1 Критический уровень (check_critical_level)

**Условия:**
- Все BUY ордера исполнены (на бирже и локально)
- `total_executed_buys >= MAX_BUY_ORDERS` (130)
- `deposit_requested == False`

**Действия:**
1. Состояние бота: `CRITICAL`
2. Флаг `deposit_requested = True`
3. Создается критическая SELL сетка:
   - 3 ордера от VWAP
   - Уровни: +2%, +4%, +6%
4. Отправляется уведомление о необходимости пополнения

### 7.2 Защита большой сетки

**Защита «3 BUY → добавить до 5 BUY внизу»** (см. `GRID_PROTECTION.md`):
- При открытых BUY ≤ 3 и общем числе открытых ордеров > порога (62 при шаге 1.5%, 127 при 0.75%) бот добавляет до 5 BUY ордеров внизу сетки (`create_buy_orders_at_bottom`), чтобы растянуть сетку вниз и снизить риск быстрого перехода в ребаланс.
- Вызывается в основном цикле: `check_protection_add_five_buy_when_three_left()`.

**Подготовка к ребалансу:** при 1 оставшемся SELL ордере отменяются 5 самых низких BUY (`cancel_last_n_buy_orders(5)`), чтобы освободить USDT для будущей рыночной покупки и новой SELL-сетки. Выполняется в `handle_sell_filled()` после синхронизации ордеров.

**Fallback в create_buy_after_sell:** если на основной цене (sell_price × (1 - grid_step_pct)) уже стоит BUY, выставляется один BUY по запасному шагу (1% при шаге 1.5%, 0.5% при 0.75%), если эта цена свободна.

### 7.3 Критическая ситуация (check_critical_situation)

**Условия:**
- Все BUY ордера исполнены
- SELL ордера не созданы
- Недостаточно баланса для рыночной покупки

**Проверка:**
```
if (open_buy_orders == 0 and 
    executed_buy_orders > 0 and 
    open_sell_orders == 0 and
    base_balance <= 0 and
    quote_balance < market_buy_amount):
```

**Действия:**
1. Отправляется критическое уведомление в Telegram
2. Указывается текущий баланс
3. Требуется пополнение или ручное создание SELL ордеров

## 🔐 Этап 9: Безопасность и защита

### 9.1 Circuit Breaker

**Назначение:** Защита от каскадных сбоев API

**Логика:**
- После 5 ошибок API → открывается circuit breaker
- Все запросы блокируются на 60 секунд
- После таймаута → переход в HALF_OPEN
- После 2 успешных запросов → закрывается

**Исключения:**
- Ошибки неверного API ключа НЕ считаются как failures
- Это ошибка конфигурации, а не проблема API

### 9.2 Rate Limiting

**Назначение:** Соблюдение лимитов биржи

**Параметры:**
- 90 запросов в минуту
- Автоматическое ожидание при превышении

### 9.3 Валидация ордеров

**Проверки перед размещением:**
- `qty >= minQty`
- `notional >= minNotional` (или 0)
- Цена округлена до `tickSize`
- Количество округлено до `stepSize`
- Цена > 0 и разумная (не слишком низкая)

## 🔄 Полный цикл работы (пример)

1. **Запуск:** Пользователь нажимает "▶️ Старт"
2. **Инициализация:** Создается сетка (5 SELL + N BUY)
3. **Ожидание:** Бот ждет исполнения ордеров
4. **BUY исполнен:**
   - Добавляется в FIFO
   - Создается SELL ордер
   - Проверяется Pyramiding
5. **SELL исполнен:**
   - Рассчитывается прибыль (FIFO)
   - Добавляется в profit_bank
   - Проверяется Pyramiding
   - Проверяется ребалансировка
6. **Pyramiding:** Если есть прибыль → создается новый BUY
7. **Ребалансировка:** Если все SELL исполнены → рыночная покупка + новые SELL
8. **Цикл повторяется:** Шаги 3-7 повторяются постоянно

## 📝 Важные особенности

### FIFO учет позиций
- Все покупки добавляются в очередь с временной меткой
- При продаже используется самая старая позиция
- Обеспечивает корректный расчет прибыли

### Автоматическое восстановление
- При перезапуске бот восстанавливает все ордера
- Продолжает работу с того же места
- Не теряет позиции и прибыль

### Гибкая настройка
- Любая торговая пара BingX
- Настраиваемый шаг сетки
- Настраиваемый размер ордера
- Масштабируемая сетка (до 130 ордеров)
